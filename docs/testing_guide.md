# Testing Guide：Escape the Equation

本文件記錄《Escape the Equation：教室危機》的主要測試項目，用於確認遊戲流程、隨機題庫、動態密碼、輸入防呆、提示系統、生命值與分數、出口門解鎖、結局統計、玩家評級，以及 GitHub Pages 線上部署版本是否正常運作。

這份文件的用途不是單純列出「測過了」而已，而是讓後續每次修改程式碼後，都可以依照固定流程重新檢查，避免改了一個小功能，結果把原本能正常運作的系統搞壞。這種事在前端專案很常見，尤其是遊戲類專案，DOM、狀態、按鈕、計時器和資料邏輯全部攪在一起時，沒測試流程很容易變成大型抓蟲現場。

## 1. 測試目的

本測試文件的目的，是確保遊戲在本機與 GitHub Pages 上都能正常執行，並確認主要功能不會因後續修改而被破壞。

本專案目前已具備多個互動系統，包含：

- 隨機題庫系統
- 動態最終密碼系統
- 輸入防呆系統
- 提示扣分與扣時間系統
- 生命值與分數系統
- 出口門鎖定與解鎖系統
- 結局統計系統
- 玩家表現評級系統
- GitHub Pages 部署展示

因此測試重點不只在於「畫面能不能打開」，還要確認每個系統之間的連動是否正確。

### 主要測試目標

本文件主要檢查以下目標：

- 遊戲可以正常啟動
- 首頁、遊戲畫面、謎題視窗與結局畫面能正常切換
- 五個教室物件可以正常互動
- 每局題目會從各自題庫中隨機抽取
- 解題後可以取得正確碎片
- 最終密碼會依照當局碎片動態生成
- 輸入防呆能正確處理不同格式
- 答錯時生命值、分數與時間扣除邏輯正確
- 提示系統只會在同一題第一次使用時扣除代價
- 出口門會在五個碎片收集後解鎖
- 結局統計能正確顯示玩家表現
- 玩家評級能依照條件正確判定
- GitHub Pages 部署版本可以正常遊玩

## 2. 測試環境

本專案目前主要以瀏覽器執行，因此測試環境分為本機測試與線上測試兩種。

### 2.1 本機測試環境

本機測試主要用於開發階段，確認修改後的程式碼能否即時正常運作。

建議測試環境：

- 作業系統：Windows
- 編輯器：Visual Studio Code
- 執行方式：Live Server
- 瀏覽器：Microsoft Edge 或 Google Chrome
- 檔案來源：本機專案資料夾

建議使用 Live Server，而不是直接雙擊 `index.html`。雖然本專案目前不依賴後端伺服器，但使用 Live Server 可以更接近一般網站執行方式，也比較方便重新整理與測試。

### 2.2 線上測試環境

線上測試主要用於確認 GitHub Pages 部署後的版本是否正常。

部署平台：

- GitHub Pages

線上網址：

```text
https://chiou-jiakang.github.io/Escape-the-equation/
```

線上測試很重要，因為本機能跑不代表部署後一定正常。常見問題包含：

- 檔案路徑錯誤
- 檔名大小寫不一致
- GitHub Pages 尚未完成部署
- README 已更新但網站尚未重新部署
- 本機檔案有改但忘記 push

## 3. 基本啟動測試

### 測試目的

確認遊戲可以正常載入首頁，並能從開始畫面切換到遊戲畫面。

這是最基本的測試。如果首頁都無法正常顯示，後面的隨機題庫、輸入防呆、結局評級全部不用測，因為遊戲根本還沒開始就爆了。

### 測試步驟

1. 使用 VS Code 開啟專案資料夾。
2. 使用 Live Server 開啟 `index.html`。
3. 確認首頁標題為「逃離方程式」。
4. 確認副標題為「教室危機」。
5. 確認首頁說明文字正常顯示。
6. 確認「進入密室」按鈕可見。
7. 點擊「進入密室」。

### 預期結果

- 首頁正常顯示。
- 文字沒有明顯錯字或破版。
- 點擊「進入密室」後，畫面切換到遊戲畫面。
- 遊戲畫面顯示倒數時間、生命值與分數。
- 初始時間應為 `10:00`。
- 初始生命值應為 3 顆愛心。
- 初始分數應為 100。
- 五個主要物件與出口門皆可見。

### 測試原因

這項測試可以確認：

- HTML 結構正常
- CSS 畫面顯示正常
- JavaScript 開始遊戲事件正常
- 起始遊戲狀態正常初始化

如果開始按鈕沒有反應，通常可能是 `script.js` 沒有正確載入，或 DOM id 與 JavaScript 選取器不一致。

## 4. 教室物件互動測試

### 測試目的

確認五個主要答題區可以正常開啟謎題視窗，且彈窗內容完整。

### 測試物件

需要測試的五個主要答題物件為：

- 黑板
- 上鎖的書桌
- 破鐘
- 座標地圖
- 方程式終端

出口門也需要測試，但出口門屬於最終密碼流程，會在後面的「出口門解鎖測試」中處理。

### 測試步驟

1. 進入遊戲畫面。
2. 依序點擊黑板、上鎖的書桌、破鐘、座標地圖與方程式終端。
3. 每次點擊後，確認謎題彈窗是否出現。
4. 確認彈窗中包含以下元素：
   - 題目標題
   - 題目內容
   - 答案輸入框
   - 提交答案按鈕
   - 使用提示按鈕
   - 關閉按鈕
5. 點擊關閉按鈕，確認彈窗可以正常關閉。

### 預期結果

- 每個物件都能正常開啟對應題目。
- 題目標題不應顯示空白。
- 題目內容不應顯示 `undefined`。
- 輸入框可以正常輸入。
- 提交按鈕與提示按鈕可以點擊。
- 關閉按鈕可正常關閉彈窗。

### 測試原因

這項測試可以確認：

- HTML 中 `data-puzzle` 屬性正確
- JavaScript 可以根據物件 key 找到對應題目
- `puzzles.js` 題庫有正常載入
- 彈窗內容能正確更新

如果點擊物件後彈窗空白，常見原因是 `puzzles.js` 沒有載入，或 `script.js` 載入順序錯誤。

正確載入順序應為：

```html
<script src="puzzles.js"></script>
<script src="script.js"></script>
```

`puzzles.js` 必須在 `script.js` 前面，否則 `script.js` 會找不到題庫資料。

## 5. 隨機題庫測試

### 測試目的

確認每次開始遊戲時，系統會從各答題區題庫中隨機抽題，而不是固定顯示同一組題目。

### 測試步驟

1. 開始第一局遊戲。
2. 點擊五個主要物件，記錄每個物件的題目內容。
3. 重新整理頁面或完成遊戲後點擊重新挑戰。
4. 開始第二局遊戲。
5. 再次點擊五個主要物件，記錄題目內容。
6. 比較兩局的題目是否可能不同。
7. 重複測試數次，觀察題目是否有變化。

### 預期結果

- 每個答題區都會從自己的題庫中抽出一題。
- 多次遊玩時，題目組合可能改變。
- 題目仍維持各自區域的數學主題：
  - 黑板應為代數與方程式
  - 書桌應為比例、百分比、折扣、濃度或分配問題
  - 破鐘應為數列與規律
  - 座標地圖應為座標幾何
  - 方程式終端應為二次方程、函數或 access key 題

### 測試原因

這項測試可以確認隨機題庫系統正常。  
如果每局題目完全一樣，可能代表：

- 隨機抽題函式沒有被呼叫
- 題庫只有一題
- 遊戲重新開始時沒有重新初始化題目
- 瀏覽器仍停留在舊版本快取

### 補充說明

隨機題庫不保證每次都一定不同。因為隨機抽題有機率抽到同一題，所以不能只測一局。建議至少測試 3 到 5 局，觀察是否有題目變化。

## 6. 動態最終密碼測試

### 測試目的

確認最終密碼會根據當局取得的五個碎片自動生成，而不是固定密碼。

### 測試步驟

1. 開始一局遊戲。
2. 解開五個主要謎題。
3. 記錄右側「已取得碎片」中的五個碎片。
4. 依照出口門規則排列：

```text
時鐘 → 地圖 → 終端 → 黑板 → 書桌
```

5. 將排列後的數字串接成最終密碼。
6. 點擊出口門。
7. 輸入組合後的最終密碼。
8. 提交答案。

### 預期結果

- 解開五題後，五個碎片會出現在右側「已取得碎片」區域。
- 出口門會從鎖定狀態變成解鎖狀態。
- 按照規則組合出的密碼可以成功通關。
- 不同局的最終密碼會因題目不同而改變。

### 測試原因

這項測試可以確認：

- 每題答案能正確成為碎片
- 碎片能正確存入 inventory
- 最終密碼能依照指定順序生成
- 出口門答案不是寫死的固定值

### 範例

假設某局取得的碎片如下：

```text
時鐘片段：36
地圖片段：13
終端片段：7
黑板片段：125
書桌片段：600
```

依照規則：

```text
時鐘 → 地圖 → 終端 → 黑板 → 書桌
```

最終密碼應為：

```text
36137125600
```

若輸入此密碼後成功逃脫，代表動態密碼系統正常。

## 7. 輸入防呆測試

### 測試目的

確認輸入格式錯誤時不會被視為真正答錯，避免玩家因輸入法、空白或符號問題受到不合理懲罰。

本專案將錯誤分成兩種：

1. 格式錯誤
2. 答案錯誤

格式錯誤不應扣生命、不應扣分、不應扣時間。  
只有格式正確但答案錯誤，才應進入答錯懲罰流程。

## 7.1 空白輸入測試

### 測試步驟

1. 開啟任一謎題。
2. 不輸入任何內容。
3. 直接按下提交答案。

### 預期結果

- 顯示提醒訊息。
- 不扣生命。
- 不扣分。
- 不扣時間。
- 答錯次數不增加。

### 測試原因

空白輸入通常代表玩家尚未作答，不應被視為真正答錯。

## 7.2 中文數字測試

### 測試輸入

```text
五
```

### 預期結果

- 顯示格式錯誤提醒。
- 不扣生命。
- 不扣分。
- 不扣時間。
- 答錯次數不增加。

### 測試原因

目前系統只接受阿拉伯數字。中文數字雖然語意上可能代表答案，但尚未設計中文數字轉換功能，因此應提示格式錯誤。

## 7.3 英文字母測試

### 測試輸入

```text
five
```

### 預期結果

- 顯示格式錯誤提醒。
- 不扣生命。
- 不扣分。
- 不扣時間。
- 答錯次數不增加。

### 測試原因

答案只需要輸入數字，不需要輸入英文單字或完整句子。

## 7.4 特殊符號與 emoji 測試

### 測試輸入

```text
5!
🙂
```

### 預期結果

- 顯示格式錯誤提醒。
- 不扣生命。
- 不扣分。
- 不扣時間。
- 答錯次數不增加。

### 測試原因

特殊符號和 emoji 不屬於數字答案，應被視為格式錯誤，而不是答錯。

## 7.5 全形數字測試

### 測試輸入

```text
５
```

### 預期結果

- 系統會將全形數字轉換為半形數字。
- 若答案正確，應判定為答對。
- 若答案錯誤，才進入答錯懲罰流程。

### 測試原因

玩家可能使用中文輸入法輸入全形數字。為了提升使用體驗，系統應支援全形數字轉換。

## 7.6 多餘空白測試

### 測試輸入

```text
2 6 4
```

### 預期結果

- 系統會移除空白後判斷。
- 若組合後答案正確，應判定為答對。
- 不應因空白而誤判。

### 測試原因

玩家可能不小心在數字之間加入空白，系統應能處理這種常見輸入問題。

## 7.7 `x=答案` 格式測試

### 測試輸入

```text
x=5
```

### 預期結果

- 系統可接受此格式。
- 系統會去除前方 `x=` 後進行判斷。
- 若答案為 5，應判定為答對。

### 測試原因

部分玩家在解方程式時會自然輸入 `x=5`。此格式應被接受，以避免使用者因表達方式不同而被誤判。

## 8. 答錯懲罰測試

### 測試目的

確認格式正確但答案錯誤時，系統會正確扣除生命、分數與時間。

### 測試步驟

1. 開啟任一謎題。
2. 輸入格式正確但不正確的數字答案。
3. 提交答案。
4. 觀察生命值、分數、時間與答錯次數。

### 預期結果

- 生命值減少 1。
- 分數減少 10。
- 剩餘時間減少 15 秒。
- 答錯次數增加 1。
- 畫面出現錯誤回饋效果。
- 彈窗產生震動效果。

### 測試原因

答錯懲罰是遊戲壓力來源之一。若答錯不扣除任何代價，玩家可以無限亂猜，遊戲挑戰性會下降。

## 9. 提示系統測試

### 測試目的

確認提示系統只會在同一題第一次使用時扣除代價，避免玩家重複點擊提示造成重複扣分。

### 測試步驟

1. 開啟任一謎題。
2. 點擊「使用提示」。
3. 記錄分數與時間。
4. 再次點擊「使用提示」。
5. 觀察分數、時間與提示次數是否再次變動。

### 預期結果

第一次使用提示時：

- 分數減少 15。
- 時間減少 20 秒。
- 提示次數增加 1。
- 顯示提示內容。

同一題再次使用提示時：

- 不再扣分。
- 不再扣時間。
- 提示次數不再增加。
- 顯示已扣過提示代價的訊息。

### 測試原因

提示是輔助機制，不是懲罰陷阱。玩家重複查看同一題提示時，不應被重複扣除代價，否則體驗會很差。

## 10. 答對與碎片收集測試

### 測試目的

確認答對題目後，系統會正確給予碎片、加分，並更新畫面狀態。

### 測試步驟

1. 開啟任一謎題。
2. 輸入正確答案。
3. 提交答案。
4. 觀察右側碎片區、分數與物件狀態。

### 預期結果

- 顯示答對訊息。
- 玩家取得對應碎片。
- 右側「已取得碎片」新增一筆資料。
- 分數增加 20。
- 對應物件呈現已解開狀態。
- 該題輸入框與按鈕會被停用，避免重複提交。
- 彈窗短暫顯示答對訊息後自動關閉。

### 測試原因

碎片收集是遊戲主線流程。如果碎片沒有正確新增，最終密碼就無法組合，遊戲會卡住。

## 11. 已解開謎題重複點擊測試

### 測試目的

確認玩家重新點擊已解開的謎題時，不會重複加分或重複取得碎片。

### 測試步驟

1. 解開任一謎題。
2. 等彈窗關閉。
3. 再次點擊同一個物件。
4. 觀察彈窗內容、分數與碎片列表。

### 預期結果

- 系統顯示該謎題已經解開。
- 顯示已取得的碎片內容。
- 不會再次加分。
- 不會再次新增碎片。
- 不會重複觸發答對流程。

### 測試原因

這項測試可以防止玩家刷分。  
如果已解題目可以重複加分，分數系統就會失去意義，很蠢但很常見。

## 12. 出口門鎖定測試

### 測試目的

確認五個主要謎題尚未解完前，出口門不能進入最終密碼輸入流程。

### 測試步驟

1. 開始遊戲。
2. 不解任何題目，直接點擊出口門。
3. 或只解開部分題目後點擊出口門。

### 預期結果

- 出口門維持鎖定狀態。
- 系統提示必須先解開五個教室謎題。
- 不允許輸入最終密碼。
- 不會提前通關。

### 測試原因

出口門必須依賴五個碎片。如果能直接輸入最終密碼，前面的解題流程就會被跳過。

## 13. 出口門解鎖測試

### 測試目的

確認五個主要謎題全部解完後，出口門會變成可使用狀態。

### 測試步驟

1. 解開五個主要謎題。
2. 觀察出口門視覺狀態。
3. 點擊出口門。
4. 確認最終密碼輸入視窗正常顯示。

### 預期結果

- 出口門解除鎖定。
- 出口門出現發光效果。
- 任務文字提示五個碎片已收集完成。
- 點擊出口門後，可以輸入最終密碼。

### 測試原因

這項測試可以確認 `allMainPuzzlesSolved()` 與 `updateDoorState()` 正常運作。  
如果門沒有解鎖，玩家即使完成五題也無法通關。

## 14. 最終密碼錯誤測試

### 測試目的

確認玩家輸入錯誤最終密碼時，系統會依照一般答錯規則處理。

### 測試步驟

1. 解開五個主要謎題。
2. 點擊出口門。
3. 輸入錯誤但格式正確的數字密碼。
4. 提交答案。

### 預期結果

- 系統判定答錯。
- 生命值減少 1。
- 分數減少 10。
- 時間減少 15 秒。
- 答錯次數增加 1。
- 不會成功逃脫。

### 測試原因

最終密碼也應遵守同一套答題邏輯，不能因為是出口門就跳過錯誤處理。

## 15. 最終密碼成功測試

### 測試目的

確認輸入正確最終密碼後，遊戲會進入成功結局。

### 測試步驟

1. 解開五個主要謎題。
2. 依照規則組合最終密碼。
3. 點擊出口門。
4. 輸入正確密碼。
5. 提交答案。

### 預期結果

- 遊戲進入成功結局畫面。
- 顯示「成功逃脫」。
- 顯示玩家評級。
- 顯示結局統計。
- 不再繼續倒數。

### 測試原因

這是完整遊戲流程的終點。若成功密碼無法通關，代表動態密碼或答案判斷流程有問題。

## 16. 結局統計測試

### 測試目的

確認遊戲結束後會顯示完整統計資料。

### 測試步驟

1. 正常完成一局遊戲。
2. 進入結局畫面。
3. 檢查結局統計區塊。

### 預期結果

結局畫面應顯示：

- 玩家評級
- 最終分數
- 剩餘時間
- 剩餘生命
- 答錯次數
- 使用提示次數
- 完成碎片數

### 測試原因

結局統計讓玩家理解自己的表現。若統計資料錯誤，評級系統也可能跟著失準。

## 17. 玩家評級測試

### 測試目的

確認不同遊戲表現會得到合理評級。

玩家評級不是只看分數，而是根據成功狀態、剩餘生命、剩餘時間與提示使用次數進行判斷。

## 17.1 Perfect Escape 測試

### 條件

- 成功逃脫
- 生命值為 3 / 3
- 使用提示 0 次
- 剩餘時間至少 300 秒

### 預期結果

```text
Perfect Escape｜完美逃脫
```

### 測試原因

Perfect Escape 是最高評級，必須代表快速、無傷、無提示通關。

## 17.2 Clean Escape 測試

### 條件

- 成功逃脫
- 生命值為 3 / 3
- 使用提示不超過 1 次
- 剩餘時間至少 180 秒
- 未達 Perfect Escape

### 預期結果

```text
Clean Escape｜俐落逃脫
```

### 測試原因

Clean Escape 代表穩定通關，但未達完美標準。

## 17.3 Narrow Escape 測試

### 條件

成功逃脫，且符合以下任一項：

- 生命值只剩 1
- 剩餘時間少於 90 秒
- 使用提示達 3 次以上

### 預期結果

```text
Narrow Escape｜驚險逃脫
```

### 測試原因

Narrow Escape 代表玩家雖然成功，但過程接近失敗邊緣。

## 17.4 Escaped 測試

### 條件

- 成功逃脫
- 但不符合 Perfect Escape、Clean Escape 或 Narrow Escape

### 預期結果

```text
Escaped｜普通逃脫
```

### 測試原因

Escaped 是成功逃脫後的剩餘分類，用來處理不屬於其他高低評級的普通通關情況。

## 17.5 Failed 測試

### 條件

- 生命值歸零
- 或時間歸零

### 預期結果

```text
Failed｜逃脫失敗
```

### 測試原因

只要生命或時間任一條件歸零，就應判定為失敗。

## 18. 重新挑戰測試

### 測試目的

確認遊戲結束後可以重新開始，並且所有狀態會重置。

### 測試步驟

1. 完成或失敗一局遊戲。
2. 在結局畫面點擊「重新挑戰」。
3. 檢查遊戲是否重新載入。

### 預期結果

- 遊戲重新開始。
- 生命值重置為 3。
- 分數重置為 100。
- 時間重置為 10:00。
- 答錯次數重置為 0。
- 提示次數重置為 0。
- 碎片清單清空。
- 題目重新隨機抽取。

### 測試原因

如果重新挑戰沒有正確重置狀態，玩家可能會帶著上一局的碎片、分數或錯誤次數進入下一局，造成邏輯錯亂。

## 19. GitHub Pages 測試

### 測試目的

確認線上部署版本可以正常遊玩，且功能與本機版本一致。

### 測試網址

```text
https://chiou-jiakang.github.io/Escape-the-equation/
```

### 測試步驟

1. 開啟 GitHub Pages 網址。
2. 確認首頁正常顯示。
3. 點擊「進入密室」。
4. 點擊五個物件。
5. 解開題目並取得碎片。
6. 組合最終密碼。
7. 完成逃脫。
8. 檢查結局統計與評級。

### 預期結果

- 網站可正常載入。
- 遊戲功能與本機版本一致。
- README 中的線上連結可正常使用。
- 不應出現 404。
- 不應出現缺少 JavaScript 或 CSS 的情況。

### 測試原因

GitHub Pages 是此專案的展示入口。若線上版本無法正常執行，即使本機版本正常，也會影響專案展示效果。

## 20. 回歸測試建議

### 測試目的

後續每次修改程式碼後，都應重新確認核心功能沒有被破壞。

### 建議回歸測試項目

每次修改後，至少應測試：

- 首頁是否正常開啟
- 是否能進入遊戲
- 五個物件是否能開啟題目
- 是否能答對並取得碎片
- 是否能使用提示
- 是否能處理錯誤輸入
- 出口門是否能正確鎖定與解鎖
- 最終密碼是否能通關
- 結局統計是否正常
- 玩家評級是否合理
- GitHub Pages 是否正常更新

### 測試原因

這些項目屬於核心流程。只要其中一項壞掉，遊戲體驗就會明顯受影響。

## 21. 已知限制

目前版本仍有以下限制：

- 尚未加入音效系統
- 尚未加入背景音樂或環境音
- 尚未加入音效開關或靜音控制
- 尚未加入最高分紀錄
- 手機版排版仍可進一步優化
- 尚未提供難度選擇模式
- 尚未提供多房間關卡
- 尚未設計正式的自動化測試

這些限制可作為後續版本的改進方向。

## 22. 測試結論

目前版本已完成主要遊戲流程與核心互動功能，包含隨機題庫、動態最終密碼、輸入防呆、提示系統、生命值與分數、結局統計與玩家評級。

後續若加入音效、最高分紀錄、難度選擇或多房間系統，應依照本文件重新進行回歸測試，確保新增功能不會破壞既有流程。
